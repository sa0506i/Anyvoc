# v2 vs v3 A/B Sweep — Slice 7b.4 Go/No-Go

**Date:** 2026-04-23
**Scope:** 132 combos per version (12 learn × 11 natives), seed=42, identical URLs.
**Cost:** ~$0.03 incremental for the v3 sweep (v2 reused from the Slice-7 run).
**Wall time:** v2 = 55.7 min, v3 = 67.0 min (+20%).

## Recommendation: **HOLD on v2 as default. v3 is a partial win; needs targeted iteration before flip.**

v3 delivers a decisive win on the primary motivating regression from the v2 sweep (verb-count collapse on noun-dense text), but regresses significantly on Scandi noun shape compliance. The net is a mixed bag — not cleanly merge-eligible as a default flip. The Scandi regression has a specific mechanism (mass-noun allowance leaking from translation rule into extraction behaviour) that can be addressed in a Slice-7c iteration.

## What v3 solved (the wins)

**Type-classification drift — the Slice-7b.2 primary motivator.** v3's symmetric CAPS-headed NOUN/VERB/ADJECTIVE/PHRASE blocks and explicit imperative-normalisation instruction restored balanced extraction on noun-dense texts.

| **Polish verbs** | 8 | **0** | 67 | +67 (∞) |

v3 extracts more verbs than even v1 — the type rebalance works. Italian carbonara (the pathological case) went from 0 verbs under v2 to 9 under v3 for native=en, though it remained at 0 for native=cs (articleless native + recipe text stays pathological — see Residual section).

**Scandi Def-Suffix Recognition quality (when present).** v3 emits 12.6% of Scandi nouns as suffix-definite vs 5.5% in v2 — the prompt restructure exposes more suffix-def cases.

## What v3 broke (the regressions)

**Scandi noun prefix compliance.** Samples of v3 bare Scandi nouns (all abstract/mass in legal-text corpus): `språk`, `samfunnsområde`, `likestilling`, `bokmål`, `nynorsk`, `status`, `ansvar`, `næring`, `konkurranse`, `vilkår`. All tagged `source_cat: "bare"` by the LLM and emitted WITHOUT the required `en/ett/ei/et` prefix. The `SCANDINAVIAN_NOUN_RULE_V3` explicitly demands the prefix on bare source — the LLM is ignoring it.

| Scandi shape | v2 share | v3 share | Δ |
|---|---|---|---|
| `en/ett/ei/et` prefix | 81.6% | 46.1% | **−35.5 pp** |
| suffix-definite | 5.5% | 12.6% | +7.1 pp |
| truly bare (rule violation) | 12.9% | **41.3%** | +28.4 pp |

**Mechanism:** v3's `buildTranslationRuleV3` carries a mass-noun allowance ("For abstract / mass / uncountable nouns whose natural dictionary form in {native} goes bare, emit bare — do NOT force an indefinite article where it would read as stilted"). Scope-wise this was intended as a TRANSLATION-side allowance, but the LLM generalised it to the source-extraction behaviour for Scandi abstract nouns. A single clause anywhere in the prompt that says "abstract nouns can go bare" seems to override the Scandi-specific "always prefix" rule when the abstract noun is Scandi.

**Translation-Target Match Rate.** 87.5% → 77.8% (−9.8 pp). Correlated with the Scandi bare-explosion: when Scandi nouns are extracted bare but still tagged `source_cat: bare`, the matrix expects the native's indef form, but the LLM often emits the native's bare form too.

**Secondary regressions.** Core-Vocab Stability −13%, Cross-Native Jaccard −3%, Wall time +20%, cost +4%. All consistent with the bare-explosion producing more cross-native divergence.

## Residual: it×cs×recipe stays pathological

Italian learning + Czech native + recipe text (the tavolartegusto carbonara URL) produced 0 verbs under v3, same as under v2. Isolated same-URL fresh runs:

- v3 it→en carbonara: 45n / 9v / 8adj / 3ph (balanced) ✓
- v3 it→cs carbonara: 50n / 0v / 0adj / 0ph (still all-noun) ✗

Hypothesis: articleless-native (cs) + noun-dense recipe text creates a Mistral interaction where the LLM simplifies the task to "list the ingredient nouns in bare Czech form". Since Czech translations are always bare, the noun-heavy translation scope may dominate the LLM's attention budget for this combo. Not a v3 regression — inherited from v2 — but not fixed either.

## Recommendation for Slice 7c

Four targeted edits for a tightened v3 / v4:

1. **Scope-fence the mass-noun allowance** so it only applies to the TRANSLATION side, never leaks into extraction. Concrete prompt edit: move the mass-noun clause out of the TRANSLATION block prose and into a separate EXCEPTION clause with tighter wording: _"Target-side exception: when the ${nativeLanguageName} dictionary form of an abstract/mass noun is bare (e.g. `freedom`, `equality`, `respect`), the translation may be bare. This DOES NOT affect the ORIGINAL field extraction — Scandi nouns still MUST carry en/ett/ei/et or the suffix-definite form even for abstract meanings."_
2. **Double-down on Scandi prefix enforcement.** Add a specific follow-up sentence after the `SCANDINAVIAN_NOUN_RULE_V3` block: _"Abstract Scandi nouns (respekt, språk, ansvar, frihet) ARE STILL subject to the en/ett/ei prefix rule. There is no exception for abstract meanings in the ORIGINAL field."_
3. **Keep the type rebalance.** That part is clearly working and delivering the intended win.
4. **Keep the strict Scandi-INDEF mirror in the translation rule.** Regressions in Translation-Target Match Rate (−10 pp) are partly collateral from the bare-explosion, not the mirror rule itself. Once the Scandi source-side is stable, the translation metric should recover.

## Decision

- `defaultPromptVersion()` stays `'v2'`.
- v3 remains opt-in via `ANYVOC_PROMPT_VERSION=v3` — available for iteration and Slice-7c testing.
- No Production changes. No rollback needed.
- Slice 7c (tightened v3 or a v4) is a future task; not blocking.

Unchanged from Slice 7: setting `ANYVOC_PROMPT_VERSION=v1` on the Fly machine reverts to the pre-Matrix-Regel baseline without redeploy. v3 opt-in adds `ANYVOC_PROMPT_VERSION=v3` for experimental users. Both are env-var only — no code change needed.
